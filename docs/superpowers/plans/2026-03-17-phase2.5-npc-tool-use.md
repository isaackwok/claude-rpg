# Phase 2.5: NPC Tool Use — Implementation Plan

## Context

Phase 2 (merged) gives NPCs streaming dialogue via the Anthropic SDK. But NPCs are still just chatbots — they can't actually do anything productive. Phase 2.5 makes NPCs real productivity agents by adding tool use: file operations, web search, and shell commands. This is the core value proposition of Claude RPG ("increase productivity in a funny way").

## Design Decisions

- **Scope**: Multi-folder — user approves directories via Notice Board or just-in-time during tool calls
- **Approval**: Every tool call requires user confirmation. Folder access uses 3-tier approval: "Add to approved" / "Allow once" / "Deny"
- **Tools**: read_file, write_file, edit_file, list_files, web_search, run_command
- **UX**: Minimal indicators in chat (status line + results in NPC text)
- **Architecture**: Separate ToolExecutor service with registry pattern
- **Folder management**: Notice Board (公告欄) — a world object in the town square. RPG metaphor: approved folders = "posted scrolls" on the board

### Multi-Folder Model: Notice Board

**Concept:** Approved directories are "scrolls" (卷軸) posted on the Notice Board (公告欄) in the town square. Any NPC can access files within approved directories (with per-action confirmation). The board is a shared town resource, not owned by any NPC.

**Two ways to add folders:**

1. **In-world (Notice Board):** Player walks to the board in town square, interacts → opens a panel showing all posted scrolls. Can add new scrolls (folder picker) or remove existing ones.

2. **Just-in-time (during tool calls):** When an NPC tries to access a path outside approved folders, the confirmation dialog offers:
   - **張貼卷軸 (Post Scroll)** — adds folder to the board permanently
   - **僅此一次 (Just Once)** — allows this single tool call without adding to the board
   - **拒絕 (Deny)** — blocks the tool call

**Data model:** `ApprovedFolder[]` persisted in `app.getPath('userData')/approved-folders.json`

```typescript
interface ApprovedFolder {
  path: string // absolute path
  label: string // user-friendly name (basename or custom)
  addedAt: number // timestamp
}
```

**Phaser integration:** New interactable object "noticeBoard" on the town map, near the Elder. Uses the same interaction pattern as NPCs (proximity zone + click).

### Step 2: Tool Definitions + ToolExecutor (backend logic)

**Files to create:**

- `src/main/tools/tool-definitions.ts` — **NEW** — Anthropic tool schemas + per-NPC tool assignment map (`AGENT_TOOLS: Record<AgentId, ToolName[]>`)
- `src/main/tools/path-utils.ts` — **NEW** — path sandboxing (resolve, realpath, traversal check)
- `src/main/tools/tool-executor.ts` — **NEW** — registry pattern with `ToolHandler` interface

**Tool handler interface:**

```typescript
interface ToolHandler {
  execute(args: Record<string, unknown>, approvedFolders: string[]): Promise<ToolExecutionResult>
}
interface ToolExecutionResult {
  success: boolean
  content: string // sent to Claude as tool_result
  summary: string // short human-readable for UI
}
```

Each handler resolves the target path and checks it falls within at least one approved folder (via `path-utils.ts`). For "Just Once" approvals, the chat handler passes the one-time path as a temporary addition to the approved list.

**Per-NPC tool assignment:**
| NPC | Tools |
|-----|-------|
| Wizard (code) | read_file, write_file, edit_file, list_files, run_command |
| Scribe (writing) | read_file, write_file, edit_file, list_files |
| Herald (comm) | read_file, write_file, edit_file, list_files |
| Commander (org) | read_file, write_file, list_files, run_command |
| Merchant (data) | read_file, write_file, list_files |
| Artisan (visual) | read_file, write_file, list_files |
| Scholar (research) | read_file, list_files, web_search |
| Elder | read_file, list_files, web_search |
| Guild Master | read_file, list_files |
| Bartender | (no tools) |

**Security constraints:**

- Path sandboxing: resolve path, check it falls within at least one approved folder via `fs.realpath` (defeats symlinks)
- File reads capped at 100KB, directory listings capped at 500 entries
- `run_command`: 30s timeout, 1MB output buffer, CWD = first approved folder (or user-specified)
- Tool confirmation timeout: 5 minutes (auto-deny)

### Step 3: Chat Handler Tool Loop (core integration)

**Files to modify:**

- `src/main/chat.ts` — refactor `Message` type to `MessageParam` (Anthropic SDK), add tool-use loop
- `src/main/index.ts` — register `chat:tool-approved` / `chat:tool-denied` IPC handlers
- `src/preload/index.ts` — add tool confirmation IPC channels
- `src/main/agents/system-prompts.ts` — add `getAgentToolContext(agentId, projectPath)` for tool-aware system prompts

**Modified `executeStream()` flow:**

1. Build tools array via `getToolsForAgent(agentId)`
2. Call `client.messages.stream(...)` with tools
3. Stream text chunks as before
4. After `stream.finalMessage()`, check `stop_reason`
5. If `'tool_use'` → enter confirmation loop:
   - Send `chat:tool-confirm` IPC for each tool_use block
   - Wait for approval/denial (Promise with 5min timeout)
   - Execute via ToolExecutor or return denial
   - Append assistant message (with tool_use blocks) + tool_result to history
   - Resume API call with updated history
   - Repeat until `stop_reason === 'end_turn'`

**History format change:** Messages become `MessageParam[]` (Anthropic SDK type) to support content blocks:

```
assistant: [{ type: 'text', text: '...' }, { type: 'tool_use', id: '...', name: 'read_file', input: {...} }]
user: [{ type: 'tool_result', tool_use_id: '...', content: '...' }]
```

**Concurrency:** Stream slot stays "active" during tool confirmation (conversation is in-progress).

**web_search handling:** Uses Anthropic's built-in type `{ type: 'web_search_20250305' }`. Not routed through ToolExecutor. Only included for NPCs in the assignment table (not all NPCs like current code).

**ConversationStatus extension:**

```typescript
| { readonly state: 'tool-confirm'; readonly toolName: string; readonly toolCallId: string; readonly args: Record<string, unknown>; readonly folderApproved: boolean }
| { readonly state: 'tool-executing'; readonly toolName: string }
```

**ToolConfirmDialog behavior:**

- If `folderApproved: true` → show [Approve] [Deny]
- If `folderApproved: false` → show [張貼卷軸 Post Scroll] [僅此一次 Just Once] [拒絕 Deny]
- "Post Scroll" calls `window.api.addApprovedFolder(folderPath)` then approves the tool call
- "Just Once" approves without adding to approved list

### Step 5: System Prompt Updates

**File to modify:** `src/main/agents/system-prompts.ts`

Add `getAgentToolContext(agentId, approvedFolders)` that appends tool instructions to the system prompt:

- Lists available tools in zh-TW with descriptions
- Lists all approved folders ("你可以存取這些已探索的領地...")
- Instructs NPC to use `list_files` first to understand project structure
- Tells NPC that each tool call requires player approval
- For `run_command`: instructs NPC to specify `cwd` from the approved folder list

When no folders are approved, tool-capable NPCs mention "到公告欄張貼卷軸" (go to the Notice Board to post a scroll).

`maxTokens` should increase from 1024 to 4096 for tool-capable NPCs (tool results can be large).

## Verification

1. **Type check**: `npx tsc --noEmit`
2. **Unit tests**: Path sandboxing (traversal attacks, symlinks, multi-folder checks), ToolExecutor handlers, FolderManager persistence, chat handler tool loop
3. **Manual E2E**:
   - Start dev (`npm run dev`)
   - Walk to Notice Board → interact → opens NoticeBoardPanel → add a folder
   - Talk to Wizard: "help me read my package.json"
   - Verify tool confirmation dialog appears with file path
   - Approve → see result incorporated in Wizard's response
   - Deny → Wizard responds gracefully
   - Talk to Wizard about a file outside approved folders → "Post Scroll" / "Just Once" / "Deny" dialog
   - "Post Scroll" → folder appears on Notice Board
   - Talk to Bartender → no tool use (chat only)
   - No folders approved → NPC says "到公告欄張貼卷軸"
4. **Security**: Attempt `../../etc/passwd` path → blocked. Symlink traversal → blocked.

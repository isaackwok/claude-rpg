# Phase 2.5: NPC Tool Use — Design Spec

## Overview

Phase 2.5 transforms NPCs from chatbots into productivity agents by adding tool use capabilities. NPCs can read/write files, search the web, and run shell commands — all with user approval. This is the core value proposition of Claude RPG: "increase productivity in a funny way."

## Goals

1. NPCs can perform real file operations, web searches, and shell commands
2. Every tool call requires explicit user confirmation (security-first)
3. Users manage approved folders through an in-world "Notice Board" metaphor
4. Tool capabilities are assigned per-NPC based on their RPG role
5. Path sandboxing prevents access outside approved directories

## Non-Goals

- Autonomous tool execution without confirmation
- Custom tool definitions (Phase 4+)
- Multi-agent tool orchestration (Phase 5)
- Background/scheduled tool execution

## Architecture

### Tool Flow

```
User message → Anthropic API (with tools) → stop_reason: tool_use
  → IPC to renderer: chat:tool-confirm
  → User approves/denies in ToolConfirmDialog
  → IPC back to main: chat:tool-approved / chat:tool-denied
  → ToolExecutor runs tool (or returns denial)
  → tool_result appended to history
  → Resume Anthropic API call
  → Repeat until stop_reason: end_turn
```

### Notice Board (公告欄) — Folder Management

The Notice Board is an in-world interactable object in the town square. It manages "scrolls" (卷軸) — approved directories that NPCs can access.

**RPG Metaphor:**

- Approved folders = scrolls posted on the town notice board
- Adding a folder = posting a new scroll
- Removing a folder = taking down a scroll
- Any NPC can read scrolls on the board (shared resource)

**Two entry points:**

1. **Direct interaction** — walk to the Notice Board, press Space, manage folders
2. **Just-in-time** — during a tool call, if the path is outside approved folders, the user gets a 3-tier choice:
   - Post Scroll (add to approved permanently)
   - Just Once (allow this single operation)
   - Deny (block)

**Persistence:** `approved-folders.json` in Electron's `userData` directory.

### Tool Definitions

Six tools available to NPCs:

| Tool          | Description              | Security                                 |
| ------------- | ------------------------ | ---------------------------------------- |
| `read_file`   | Read file contents       | Path sandboxed, 100KB cap                |
| `write_file`  | Write/create file        | Path sandboxed                           |
| `edit_file`   | Find-and-replace in file | Path sandboxed                           |
| `list_files`  | List directory contents  | Path sandboxed, 500 entry cap            |
| `web_search`  | Search the web           | Anthropic built-in, no sandboxing needed |
| `run_command` | Execute shell command    | 30s timeout, 1MB output, CWD restricted  |

### Per-NPC Tool Assignment

Tools are assigned based on each NPC's RPG role and specialization:

- **Wizard (code):** read_file, write_file, edit_file, list_files, run_command — full dev toolkit
- **Scribe (writing):** read_file, write_file, edit_file, list_files — document editing
- **Herald (comm):** read_file, write_file, edit_file, list_files — message drafting
- **Commander (org):** read_file, write_file, list_files, run_command — task execution
- **Merchant (data):** read_file, write_file, list_files — data analysis
- **Artisan (visual):** read_file, write_file, list_files — design file access
- **Scholar (research):** read_file, list_files, web_search — research focus
- **Elder:** read_file, list_files, web_search — guidance + search
- **Guild Master:** read_file, list_files — read-only overview
- **Bartender:** (no tools) — chat only, as befits a bartender

### Security Model

1. **Path sandboxing:** All file paths are resolved via `fs.realpath()` and checked against approved folders. This defeats:
   - `../` traversal attacks
   - Symlink-based escapes
   - Path normalization tricks

2. **User confirmation:** Every tool call shows the user what will happen and requires explicit approval. No exceptions.

3. **Resource limits:**
   - File reads: 100KB max
   - Directory listings: 500 entries max
   - Shell commands: 30s timeout, 1MB output buffer
   - Tool confirmation: 5-minute timeout (auto-deny)

4. **No ambient authority:** Even within approved folders, each individual tool call requires confirmation.

## Data Model

### ApprovedFolder

```typescript
interface ApprovedFolder {
  path: string // absolute path to directory
  label: string // user-friendly name (defaults to basename)
  addedAt: number // Date.now() timestamp
}
```

Persisted as JSON array in `{userData}/approved-folders.json`.

### Extended ConversationStatus

```typescript
type ConversationStatus =
  | { state: 'idle' }
  | { state: 'waiting' }
  | { state: 'streaming' }
  | { state: 'error'; error: string }
  | {
      state: 'tool-confirm'
      toolName: string
      toolCallId: string
      args: Record<string, unknown>
      folderApproved: boolean
    }
  | { state: 'tool-executing'; toolName: string }
```

### Message Format Change

History transitions from simple `{ role, content }` to Anthropic SDK's `MessageParam[]` to support tool_use/tool_result content blocks.

## IPC Channels (New)

| Channel               | Direction       | Type   | Payload                                           |
| --------------------- | --------------- | ------ | ------------------------------------------------- |
| `folders:get-all`     | renderer → main | invoke | → `ApprovedFolder[]`                              |
| `folders:add`         | renderer → main | invoke | `(path: string)` → `ApprovedFolder`               |
| `folders:remove`      | renderer → main | invoke | `(path: string)` → `void`                         |
| `folders:select-add`  | renderer → main | invoke | → `ApprovedFolder \| null`                        |
| `chat:tool-confirm`   | main → renderer | send   | `ToolConfirmPayload`                              |
| `chat:tool-approved`  | renderer → main | send   | `{ agentId, toolCallId, addToApproved?: string }` |
| `chat:tool-denied`    | renderer → main | send   | `{ agentId, toolCallId }`                         |
| `chat:tool-executing` | main → renderer | send   | `{ agentId, toolName }`                           |

## Dependencies

- Phase 2 (Agent Conversations) — merged, provides streaming chat infrastructure
- No new npm dependencies required
- Anthropic SDK tool use API (already available in `@anthropic-ai/sdk`)
